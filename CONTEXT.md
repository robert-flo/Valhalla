# RaVN Installer Domain

## Core terms

### RaVN component

A user-facing resource intentionally added or maintained by the RaVN project. RaVN components are installed and tested by the independent RaVN installer.

### Independent RaVN installer

The installer responsible only for RaVN components. It is developed and tested independently from the upstream installer and does not replace upstream installation steps by default.

### Upstream installer

The existing installer that remains the source of truth for upstream installation behavior until a later migration explicitly replaces one of its sections.

### Progressive replacement

The future process of replacing individual upstream installer sections only after their equivalent RaVN component flow is complete, tested, and accepted.

### RaVN layer

The set of changes made on top of the repository baseline commit `2f94ca75e584977a9ca2e46369ec934bb1ac7ccf`. The RaVN layer is the ownership boundary for resources introduced or changed by this project.

### Resource precedence

When the RaVN layer and the upstream layer declare the same resource, the RaVN version is installed with precedence over the upstream version. This applies to documents, shell configuration, terminal configuration, and other user-facing resources.

### Baseline anchor

The baseline commit is immutable reference data for identifying the origin of changes. It is not, by itself, the installation manifest.

### Section manifest

A versioned declaration of the RaVN-owned resources and dependencies for one installation section. Section manifests are the source of truth for what is installed, verified, cleaned, or restored. Repository diffs are supporting discovery and audit data only.

### Category isolation

Each installation category owns a separate manifest and lifecycle. Installing, testing, cleaning, or rolling back one category must not implicitly modify resources owned by another category.

### Resource policy

Each manifest entry chooses its own resource policy according to the existing restore semantics. Policies are decided case by case; overwrite is expected to be common for RaVN-owned resources, but it is not an unconditional rule.

### RaVN backup

RaVN reuses the established restore and backup behavior, but writes its backups below a dedicated RaVN backup root. RaVN backups are isolated from upstream backups while retaining the same restore semantics.

### Minimal adaptation

The independent installer reuses the upstream restore implementation wherever possible. Adaptations are limited to selecting RaVN-owned manifests and identifying RaVN backup output; it does not introduce a second parser, policy language, path resolver, or restore model.

### Installation category

An independently operated group of RaVN-owned resources with its own manifest and lifecycle actions: install, test, clean, and rollback. Initial categories are desktop launchers, binaries, configurations, and applications.

### Aggregate installation

The global installation action runs categories in their defined order, records each category's result, continues after an individual failure, and presents a final per-category summary.

### Installation run

A single invocation of the installer that groups category results and their backups under one timestamped identity. A global rollback targets one installation run and restores participating categories in reverse order.

The installation run uses the existing timestamp and backup identity generated by the upstream restore flow. RaVN does not introduce a second run-ID mechanism.

### Package rollback

For package dependencies, rollback means uninstalling packages installed by the targeted RaVN installation run. File rollback restores the backups created by the restore flow; package rollback does not restore files.

Package rollback is conservative: it may remove only explicitly declared packages that RaVN queued and successfully installed. It never removes transitive dependencies or packages already present before the run, prioritizing a bootable host system.

### Launcher workflow precedent

The completed Desktop launcher workflow is the behavioral precedent for every future category. New categories reuse its menu structure, manifest audit, counts, confirmations, cleanup, rollback boundaries, and shared visual language rather than introducing a separate interaction model.

The existing launcher implementation is the concrete reference for category tests and audits; future categories must adapt that behavior rather than inventing a second audit model.

### Orchestrator boundary

The independent installer coordinates menus, category selection, manifests, shared output, and lifecycle reporting. Existing functional category implementations remain responsible for their own installation work and are invoked rather than rewritten.

### Minimal verification

Verification stays intentionally small: read-only audits, destructive file tests against a temporary home, and package dry-runs. The installer does not introduce a new transaction or snapshot system beyond the existing restore and backup behavior.

### Repository precedent

When a design question arises, the existing installer and its dependent scripts are the authoritative precedent. RaVN adapts established repository behavior and introduces new logic only when no existing behavior covers the requirement.

### Incremental category growth

Categories are introduced and expanded incrementally from resources already present in the repository. No complete category inventory or fixed migration order is assumed in advance; each new category is agreed, implemented, tested, and made reversible before the next expansion.

### Installer menu

The main installer menu exposes Desktop launchers, Binaries, Configurations, Applications, and Install everything. Future categories may be visible before implementation, but must report their unavailable status and make no changes.

### Applications

The installation category for system packages and programs managed through the existing package installation flow and its package manifest. It does not include launcher artifacts, user binaries, or configuration files.

### Binaries

RaVN-owned executable files installed into the user's `${HOME}/.local/bin`. Scripts used internally by the installer are not Binaries unless they are explicitly declared as user-installed resources.

### Configurations

RaVN-owned user configuration resources represented with the same directory layout and restore semantics as the repository's existing configuration tree. The RaVN configuration tree is a parallel overlay; it does not introduce new destination conventions or a second configuration parser.
